# ADR 009 — Recompute derived state on import (`inCheck`, PGN replay)

## Status

Accepted

## Context

`chess.codec` accepts game state from three external formats: FEN, JSON, and PGN. Each format can be hand-edited or generated by another tool, so the bytes on the wire cannot be trusted to be internally consistent with `GameState`'s invariants:

- **FEN** does not record an "in check" flag at all — it has to be derived from the placement and active color.
- **JSON** has an `inCheck` field (because it's a structural derivation of `GameState` via `zio-json`'s auto-derivation), but a hand-edited or stale snapshot can claim `inCheck = false` for a position whose king is actually attacked.
- **PGN** stores moves as SAN text. Nothing in the file format guarantees that the moves are legal — a malicious or buggy producer can record a move that violates the rules.

Two failure modes follow if we trust the wire format:
1. The TUI/GUI renders a board where the checked king isn't highlighted (or vice versa), because `inCheck` lies.
2. A loaded game lets the next move be played from an unreachable position, silently corrupting the rest of the session.

## Decision

Treat every imported game as untrusted input. Anything that can be re-derived from the canonical fields (board placement, active color, move list) **must** be re-derived during import — the wire copy is discarded.

Concretely:

- **`FenBuilder`** computes `inCheck` via `MoveValidator.isInCheck` rather than reading it from the FEN string (FEN doesn't carry it anyway, but the same rule applies to any future format that does).
- **`JsonCodec.given JsonDecoder[GameState]`** decodes through a private `GameStateDTO` and overwrites the `inCheck` field with `MoveValidator.isInCheck(dto.board, dto.activeColor)`. The wire `inCheck` is intentionally ignored.
- **`PgnParser`** does not store the SAN move list as data. It replays each parsed move through `MoveParser.parse` + `Game.applyMove`, accumulating the resulting `(Move, GameState)` pairs. If any move is illegal under `Game`, the whole import fails with `GameError.ParseError`. The output is a fully validated history that `loadGame` can drop straight into a `GameSnapshot` for undo/redo.
- **JSON `halfmoveClock` / `fullmoveNumber`** are decoded as `Option[Int]` and default to `0` / `1` when absent. This is the same "trust the structural fields, recompute or default the rest" rule applied to backwards compatibility.

## Consequences

**Benefits:**
- Imported positions are always self-consistent. The TUI's checked-king highlight and the GUI's move-legality checks see the same `GameState` invariants regardless of where the game came from.
- PGN import doubles as a regression test for the rule engine — if `Game.applyMove` rejects a previously-legal sequence, the failure surfaces at import time instead of corrupting state later.
- The same "untrusted input" boundary holds for any future format (e.g. UCI position strings, lichess JSON), so the rule scales without rewriting decoders.
- `Game.applyMove` remains the single source of truth for what a "valid game state transition" means. Codecs depend on it; codecs don't reimplement it.

**Trade-offs:**
- PGN import is O(n) in the number of moves. For typical chess games (~80 half-moves) this is negligible; for million-move synthetic test files it would matter, but no such use case exists in this project.
- `chess.codec` depends on `chess.model.rules` (for `MoveValidator` and `Game`). Codecs are no longer pure translation — they pull in the rules engine. Accepted because the alternative (a separate "rules-aware import" layer) would just move the dependency without removing it.
- Loading a game whose move list was generated by a stricter rule engine (e.g. one that rejects an obscure draw claim) will fail to import. This has not happened in practice and is preferable to silently dropping the offending move.
